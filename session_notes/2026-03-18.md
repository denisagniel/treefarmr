## Critical Bug Discovery: Parallel Execution Deadlock

**Status:** ✅ FIXED

### Problem

While attempting to benchmark thread-safety performance with the optimizing-r skill, discovered that ALL parallel executions with `worker_limit > 1` hung indefinitely. The previous stress testing only used `worker_limit = 1, 2, 4` but likely had very short run times that didn't expose the hang.

**Symptoms:**
- Any `fit_tree()` with `worker_limit > 1` hung forever (100% CPU, no progress)
- Happened on even tiny problems (100 × 10)
- First benchmark run took over 15 minutes before being killed

### Root Causes

**1. Data Race** (`src/optimizer.hpp:114`)
```cpp
bool active = true; // Not thread-safe!
```
Multiple threads reading/writing without synchronization.

**2. Periodic Termination Check** (`src/optimizer.cpp:204`)
Worker 0 only checked termination condition every 10,000 iterations. When queue emptied between checks, all workers spun forever waiting for flag update.

**3. Incorrect Logic**
Previous logic: `active = ... && (worker_limit > 1 || queue.size() > 0)`
With multiple workers, this ignored queue.size() and caused infinite spin.

### The Fix

1. **Thread-safe flag**: `bool active` → `std::atomic<bool> active`
2. **Every-iteration check**: All workers check termination every iteration:
   ```cpp
   bool should_continue = !complete() && !timeout() && state.queue.size() > 0;
   return should_continue;
   ```
3. **Local decisions**: Each worker checks queue size directly, no waiting

### Performance Results (After Fix)

Ran comprehensive benchmarks across 4 problem sizes:

**Tiny (100 × 10):**
- 2 workers: 0.71x speedup (36% efficient) - **SLOWDOWN**
- 4 workers: 0.87x speedup (22% efficient) - **SLOWDOWN**

**Small (300 × 15):**
- 2 workers: 1.17x speedup (59% efficient) - Moderate
- 4 workers: 1.19x speedup (30% efficient) - Poor

**Medium (800 × 25):**
- 2 workers: 1.25x speedup (62% efficient) - Moderate
- 4 workers: 1.44x speedup (36% efficient) - **BEST CASE**

**Large (1500 × 35):**
- 2 workers: 0.98x speedup (49% efficient) - **SLOWDOWN**
- 4 workers: 1.05x speedup (26% efficient) - Poor

### Key Finding

**Thread overhead dominates for all problem sizes tested.**

Best case is medium problems (800 × 25) with 4 workers: 1.44x speedup (36% efficient). Even this is far below ideal scaling (would be 4.0x at 100% efficiency).

### Recommendation

**Default should be `worker_limit = 1`** (single-threaded)

Reasons:
- Overhead dominates for small/large problems
- Even best case achieves only 36% efficiency
- Many problems see slowdowns with parallelization
- Users who benefit can opt-in by setting `worker_limit = 2`
