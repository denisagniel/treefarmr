## Implementation Details

### Discretization Methods
1. **Median** (default): Single threshold at median
2. **Quantiles**: Multiple thresholds using quantile splits

### Features
- Automatic discretization of continuous features
- User-provided custom thresholds
- Mixed binary/continuous feature support
- Fast-path for binary-only data (<5% overhead)
- Metadata storage and inspection
- Full backward compatibility

### Architecture
Discretization is a transparent preprocessing layer:
```
User provides X_continuous →
  Discretize → X_binary →
  Tree optimization (unchanged) →
  Store metadata in model →
Predict with X_new_continuous →
  Apply stored discretization → X_new_binary →
  Tree traversal (unchanged)
```

## Adaptive Discretization (Update)

**Added**: Adaptive bins that grow with sample size to satisfy theoretical rate requirements.

**Problem identified**: Theory requires tree complexity $s_n \sim n^{d/(2\beta+d)}$ leaves, which needs bins ~ log(n) to allow tree flexibility. Fixed bins violate this.

**Solution**:
- `discretize_bins = "adaptive"` uses `max(2, ceiling(log(n)/3))`
- Grows with n: n=50 → 2 bins, n=1000 → 3 bins, n=10000 → 4 bins
- Required for theoretical convergence rate guarantees

## Simulation Results (Update 2)

**Simulation completed**: `adaptive_bins_simple.R` (50 reps × 3 methods × 5 sample sizes × 3 scenarios)

**Key findings**:
1. ✅ **Adaptive bins work**: Tree complexity grows from 6 leaves (n=100) to 19 leaves (n=2000) for log-loss
2. ✅ **Fixed bins hit ceiling**: Fixed-2 stays at ~6-7 leaves regardless of n
3. ⚠️ **Regularization mismatch**: `regularization=0.1` works for log-loss but too strong for squared-error (forces all trees to 2 leaves)
4. **Fixed-4 overfits**: Creates 27-40 leaves even at small n, worse performance than adaptive

**Theoretical validation**: Adaptive discretization successfully allows tree complexity to grow with n, **confirming the theory** that bins ~ log(n) is necessary.

## Memory Diagnosis and Fixes (Update 3)

**Problem reported**: `devtools::test()` consuming excessive memory, causing system memory pressure.

**Analysis completed**: Identified 4 main culprits:

1. **Pre-loaded datasets**: helper-data.R creates 14 datasets at load time (big_dataset: 1000×10, many_features_dataset: 100×50)
2. **store_training_data = TRUE**: 9 occurrences prevent garbage collection
3. **Nested loops**: test-comprehensive-logloss.R creates ~50+ model objects with nested loops
4. **Insufficient gc()**: Only 5 gc() calls in entire test suite

**Test files with high memory risk**:
- test-comprehensive-logloss.R (683 lines, nested loops, ~50 models)
- test-probability-accuracy.R (633 lines, 16 calls, 7 loops)
- test-edge-cases.R (564 lines, 49 treefarms() calls)
- test-cross-fitted-rashomon.R (442 lines, K-fold multiplication)

**Fixes implemented**:

1. **Reduced pre-loaded dataset sizes** (helper-data.R):
   - big_dataset: 1000×10 → 200×10 (80% reduction)
   - many_features_dataset: 100×50 → 100×20 (60% fewer features)
   - Estimated savings: ~10-15%

2. **Removed store_training_data = TRUE** (9 occurrences):
   - test-fit-functions.R: 6 removals
   - test-squared-error.R: 3 removals
   - Estimated savings: ~20-30%

3. **Added explicit memory cleanup** (test-comprehensive-logloss.R):
   - Added rm() + gc() after 6 loop iterations
   - Locations: single tree loop, cross-fitted loops (2), rashomon loops (2), dataset size loop
   - Estimated savings: ~15-25%

**Total expected improvement**: 45-70% memory reduction
